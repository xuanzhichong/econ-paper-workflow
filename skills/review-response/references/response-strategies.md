# Response Strategies for Empirical Economics and Agricultural Economics

Choose a response strategy based on the substance of the referee point, not just on tone.

A good response strategy should match:

- the seriousness of the referee concern
- whether the concern is valid
- whether the manuscript was actually revised
- whether the requested change is feasible
- whether implementing the request would alter the estimand, weaken identification, or move the paper outside its scope

The goal is not to sound agreeable at all times.  
The goal is to respond clearly, truthfully, and traceably.

---

## Core Rule

For each referee comment, first decide:

1. Is the referee substantively right?
2. Does the manuscript need to change?
3. Can the requested change be implemented credibly?
4. If not, what is the narrowest honest accommodation?

Only after answering those questions should you choose the response strategy.

---

## 1. Accept and Revise

Use when the referee identifies a real weakness, omission, or ambiguity that should be corrected in the manuscript.

Typical cases:

- unclear identification logic
- missing robustness check
- weak exposition
- incomplete literature positioning
- missing table note, appendix material, or sample explanation

### Template logic

- acknowledge the point
- say clearly that you agree
- describe exactly what changed
- name the new table, appendix, or section
- provide the location

### Working pattern

- "We agree that ..."
- "In response, we now ..."
- "We revised Section X / added Table A.X / expanded Appendix Y ..."
- "The revised discussion appears in ..."

---

## 2. Clarify and Revise Exposition

Use when the core logic of the paper is unchanged, but the referee is right that the exposition was too compressed, confusing, or insufficiently transparent.

Typical cases:

- the design is already valid, but poorly explained
- the sample construction is in the paper, but hard to follow
- the mechanism boundary is present, but not clearly stated
- the manuscript needs clearer signposting rather than new analysis

### Template logic

- acknowledge the confusion
- state that the core interpretation remains unchanged
- explain the intended logic
- say where the text was rewritten

### Working pattern

- "Our interpretation remains unchanged, but we agree that the original exposition was too compressed."
- "We have revised the text to explain ..."
- "We now clarify this point in Section X and add a cross-reference to Appendix Y."

---

## 3. Clarify Without Revising

Use when the concern can be answered directly in the response, but no substantive manuscript revision is necessary or appropriate.

Typical cases:

- the manuscript already states the relevant point clearly enough
- the referee requests something outside the paper’s scope, but the boundary can be explained
- the issue is best resolved through a direct explanation rather than a manuscript-side change

### Template logic

- acknowledge the concern
- explain the logic directly
- state clearly that no substantive manuscript revision was made
- avoid implying that a revision occurred if it did not

### Working pattern

- "We appreciate this comment. Our reading of the issue is as follows: ..."
- "We do not make a substantive manuscript change on this point because ..."
- "To avoid overstating the revision, we address the issue here through clarification."

---

## 4. Partially Accommodate

Use when the request is reasonable but only partly feasible.

Typical cases:

- the referee asks for an extension that is useful but only partially implementable
- the data do not support the full requested exercise
- the requested analysis would be too expansive, but a narrower check is feasible
- a manuscript revision can address the core concern without fully satisfying the original request

### Template logic

- explain what was added
- explain what could not be done
- connect the partial change to the original concern
- make the remaining limitation explicit

### Working pattern

- "We agree with the concern and have incorporated part of the requested analysis."
- "Specifically, we now provide ..."
- "We do not implement the full request because ..."
- "We now state this limitation directly in the manuscript."

---

## 5. Decline with Justification

Use when the requested change would be inappropriate, impossible, or misleading.

Typical cases:

- implementing the request would change the estimand
- implementing the request would weaken identification
- the data needed are unavailable
- the requested extension falls outside the paper’s scope
- the suggested revision would make the interpretation less accurate, not more accurate

### Template logic

- respectfully explain why
- make the economics reason explicit
- note any clarification added to the manuscript
- avoid defensive or dismissive language

### Working pattern

- "We understand the reviewer’s suggestion, but we do not adopt it because ..."
- "Implementing this change would alter the estimand / weaken identification / require unavailable data."
- "We have instead clarified this scope condition in the manuscript and added a short discussion of the tradeoff."

## Good Strategy Mapping

### Identification concern
Preferred responses:

- new design defense
- new targeted check
- clearer statement of identifying assumptions
- narrower causal language if needed

### Robustness concern
Preferred responses:

- targeted sensitivity analysis
- alternative coding or sample check
- placebo or falsification exercise tied to the specific threat

### Mechanism concern
Preferred responses:

- added supporting evidence
- clearer distinction between main effect and mechanism evidence
- softened language if mechanism is not separately identified

### Exposition concern
Preferred responses:

- rewritten text
- clearer signposting
- new table note or appendix pointer
- clarified sample or design logic

### Literature-positioning concern
Preferred responses:

- add missing references
- narrow contribution claim
- distinguish direct evidence from background literature

### Sample-construction concern
Preferred responses:

- add clearer sample pipeline
- clarify exclusions, merge logic, or missing-data handling
- add appendix note or flow explanation

### Causal-language concern
Preferred responses:

- trim over-strong language
- align abstract, results, and conclusion with design strength
- distinguish suggestive evidence from stronger causal claims

### Appendix / extra-table request
Preferred responses:

- add appendix table if it directly addresses the concern
- explain if a lighter accommodation is sufficient
- avoid adding decorative appendix material with no clear role

---

## Common Strategy Errors

Avoid the following:

- agreeing too broadly when only a partial revision was made
- claiming "we revised" when the response only clarifies
- adding a check that does not actually address the concern
- declining a suggestion without explaining the estimand, identification, data, or scope reason
- sounding defensive or dismissive when refusing a request
- over-promising what a new robustness check establishes
- failing to state where the manuscript actually changed

---

## Traceability Rule

Regardless of strategy, a strong response should make clear:

1. what the concern is
2. what stance you take
3. what changed, if anything
4. where that change appears
5. what remains outside the paper’s scope, if relevant

If those five points are not easy to identify, the strategy has not yet been executed well.
